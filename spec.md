# Windmill SAO

Windmill SAO (https://github.com/kantoniak/windmill-sao) is a Simple Add-On/Shitty Add-On for electronic conference badges that uses SAO connector for 3V3 power and for I2C communication with the board. Branch https://github.com/kantoniak/windmill-sao/tree/kantoniak/rev1-rc2-wip has some additional work, notably finished model (model creation code needs some cleanup), and firmware for driving servo through PWM interface. Your task will be to provide programming guide and drop-in implementation of the rest of the firmware spec below.

## Overview

Windmill SAO is an addon that resembles a Dutch Tower windmill from mid-XIX century with rotating windsails.
* The board works with 3V3 only.
* The model contains a PCB build around CH32V003F4P6 (TSSOP) which drives the winndsails with SG90 servo.
* Servo is driven from a PWM signal generated by the chip (see the code). It's return path is a separate net that contains a 0.5 Ohm sensing resistor. R1 is connected to the uC through an RC filter.
* The main communication interface is I2C, part of the SAO connector. There is also a QWIIC passthrough on the same lines, so they share the I2C pullups. The board theoretically can be powered from QWICC connector but no one tried so far :)
* There is also UART interface, but currently unused.
* The board has a neopixel (WS2812B-2020) connected to SPI interface pins. I haven't tried running it yet.
* There are 3 pads for R0 resistors for configuring options. The options currently are not implemented.

The pinout is fixed because the boards have been produced already. Pin assignments are as follows:
* `PD0`: SWIO
* `PD7`: NRST
* `PC4`: SERVO_PWM
* `PA2`: SERVO_SENSING (connected to the high side of R1 through 1k resistory of the RC filter)
* `PC1`: SDA
* `PC2`: SCL
* `PD5`: UART_TX
* `PD6`: UART_RX
* `PC6`: LED_DATA (to DIN on the neopixel through 22 Ohms)
* `PC7`: CONFIG_A
* `PD2`: CONFIG_B
* `PD3`: CONFIG_C

## Firmware

The intented firmware architecture is as follows. The program is buit around interrupt events that drive the behavior. There are two main phases: initialization phase and event handling loop phase. Once the board runs initial configuration, it awaits events that required action. Events result in interrupts, which in sets the flags that will be handled in the main loop. Main loop sleeps once it handles the events, awaiting new interrupts.

## I2C

I2C is the main communication protocol with the board. Everything that user wants to know or influence about the board should be handled via I2C commands. The final report should document any virtual I2C registers to are created as a result of this spec. uC should use DMA to communicate with the I2C peripheral.

## Board identification

The badge addon should implement descriptor possibly similar to https://badge.team/docs/standards/sao/binary_descriptor/ Users can read this memory to get the badge details. This part of the memory is not editable.

## Servo

Servo is driven by the I2C commands that set the direction (clockwise/counter-clockwise) and the speed. Speed is from a range (please see the code). User can also send an array of speeds that will used to build a series of values for the servo (a whole array sent to the peripheral). The PWM timer should be driven using DMA. User should be able to pause or start the windsail.

I possible, I would like to let users control the loop mode, for example to allow playing sequence forwards and the backwards for smooth looping.

Please reserve more addresses for the servo options, I plan to add things like dead zone calibration settings etc.

## Servo current sensing

There should be a command that lets the users check an estimatedd current draw based on the sensing resistor. This computation happens in the main loop after the command is sent.

## Neopixel

Neopixel should be driven from the SPI port using the https://github.com/cnlohr/ch32fun/blob/master/extralibs/ws2812b_dma_spi_led_driver.h library. Again, user can send I2C commands to change the color immediately.

## UART

UART is a supplementary part of this project and has lowewr priority among design choice. For now it's enough if UART accepts `ping` and `info` commands

## Demo mode
This project is a SAO and should be attractive event without any commands sent. If CONFIG_A is not shorted to ground when the board wakes up, then The windmill behaves like a clock, doing one fact clockwise "jump" and staying idle for the rest of the second. Sending any write-type command should disable the demo mode.
